## Context

The design team requires graph mode (deterministic state machine) and agent-loop mode
(ReAct-style LLM-driven) to coexist with arbitrary bidirectional nesting:
graph → agent-loop → graph → agent-loop → … at any depth.

The mechanism is interrupt-based nesting: parent suspends at a node, hands control to child,
resumes when child returns. One `SuspendSignal` primitive covers both modes; the `Orchestrator`
owns the catch/checkpoint/dispatch/resume loop. The same `Checkpointer` SPI spans three
durability tiers: in-memory (W0) → Postgres-backed (W2) → Temporal child workflow (W4).

Strategic constraint (locked 2026-05-12): Spring AI Alibaba is a direct competitor. SAA
artifacts were studied as design references (StateGraph, BaseCheckpointSaver, interrupt()) but
were never added as Maven dependencies. `ApiCompatibilityTest` enforces this via ArchUnit.

## Architecture invariants

The `Run` entity, `Checkpointer` SPI, and `SuspendSignal` primitive are **identical** across all
three durability tiers. W2 adds a Postgres-backed `Checkpointer` impl and a Postgres-backed
`Orchestrator`; W4 routes long-TTL runs to Temporal `executeChildWorkflow`. No schema change,
no new entity, no migration of the nesting model.

## Bug fixed in C34

`IterativeAgentLoopExecutor`: when iter 0 suspends immediately (without producing output),
the initial context (`def.initialContext()`) was being saved as `RESUME_STATE_KEY`. On resume,
this caused `savedState + childResult` = `"{}child-done"` instead of `"child-done"`, breaking
the parent reasoner's discriminator check. Fix: save `RESUME_STATE_KEY` only when `i > 0`
(prior iterations produced accumulated output). `NestedDualModeIT` was already correct (L2 loop
suspends at iter 1, not iter 0). `RunStatusTransitionIT` was the failing case; now green.
