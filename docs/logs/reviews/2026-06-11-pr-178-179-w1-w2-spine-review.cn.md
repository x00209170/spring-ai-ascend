### P2: 手写 `JwtTenantValidator` 需要重新对齐“优先复用开源安全组件 API”的架构理念

位置：

- `agent-runtime/src/main/java/com/huawei/ascend/runtime/boot/JwtTenantValidator.java:13-97`
- `agent-runtime/pom.xml` 已包含 `spring-boot-starter-security` 与 `spring-boot-starter-oauth2-resource-server`
- `docs/adr/locked/0010-spring-security-oauth2.md` 明确身份基线是 OIDC / JWKS / Keycloak / BYO IdP
- `architecture/docs/L1/agent-service/ARCHITECTURE.md` 仍描述 `JwtDecoderConfig`、JWKS-backed、RS256/JWKS 方向

PR 179 当前实现了一个 JDK+Jackson 的 HS256 JWS validator：手动拆 token、手动解析 header/claims、手动 HMAC 校验、手动 exp 处理。ADR-0161 授权的是“抽出一个共用 validator，避免 runtime 和 service edge 两份拷贝”；它没有充分说明为什么要绕开现有 Spring Security / Nimbus `JwtDecoder` 生态，也没有把 HS256 手写实现限定为 dev/local fallback。

这与“尽可能复用开源组件 API”存在张力。尤其这是认证/租户隔离边界，不是普通 JSON 解析工具类。手写 JWT parser/verifier 的长期风险包括：

- OIDC/JWKS/BYO IdP 路线被短路成 shared-secret HS256。
- `nbf`、`iss`、`aud`、`kid`、key rotation、JWKS refresh 等成熟资源服务器能力缺位。
- 安全行为散落在自研代码里，后续审计需要重新证明，而不是继承 Spring Security / Nimbus 的标准语义。

建议修复：

1. 用 Spring Security / Nimbus `JwtDecoder` 负责 JWS/JWT 校验；平台代码只做 `tenant_id` claim 与 `X-Tenant-Id` / query tenant 的 cross-check。
2. 如果 W1 必须先保留 HS256，本 PR 需要把它明确标为 dev/local 或 transitional fallback，并在 ADR-0161 或新 ADR 中说明为什么临时绕开 OIDC/JWKS，以及迁移到 JWKS-backed decoder 的 trigger。
3. 测试应覆盖 `nbf`、missing `exp` 是否允许、issuer/audience 策略、key rotation / `kid` 行为，或者明确这些由 Spring Security 组件承接。

### P1: `IdempotentRequestHandler` 对 `A2AError` 失败不释放 claim，会永久阻塞同一 `messageId` 重试

位置：

- `agent-runtime/src/main/java/com/huawei/ascend/runtime/engine/a2a/IdempotentRequestHandler.java:75-88`
- 测试缺口：`agent-runtime/src/test/java/com/huawei/ascend/runtime/engine/a2a/IdempotentRequestHandlerTest.java:79-92`

实现当前逻辑：

```java
try {
    EventKind result = delegate.onMessageSend(params, context);
    ...
} catch (RuntimeException e) {
    store.release(tenantId, messageId);
    throw e;
}
```

`RequestHandler.onMessageSend(...)` 本身声明 `throws A2AError`。如果 delegate 在 acquired claim 后抛出 A2A SDK 的 `A2AError`，当前 catch 不会执行，`store.release(...)` 不会发生。后续同 tenant + same `messageId` 重试会命中 `IN_FLIGHT`，返回 “duplicate message/send is still executing”，形成不可恢复的假 in-flight 状态。

现有测试只覆盖了 `RuntimeException`：

```java
when(delegate.onMessageSend(any(), any()))
        .thenThrow(new RuntimeException("transient"))
        .thenReturn(task("task-1", TaskState.TASK_STATE_COMPLETED));
```

这没有覆盖 A2A SDK 的 checked/domain error。

影响：

- 客户端使用 idempotency key 重试失败请求时，某些合法失败路径会被永久拒绝。
- 对外表现像并发重复请求，但实际原始请求早已失败。
- 这会破坏 PR 声称的 W1 idempotency / retry contract。

建议修复：

1. 在 acquired claim 后同时 catch `A2AError` 并 release 后 rethrow。
2. 或使用 `finally`，但要避免成功完成后又 release 已 complete 的 reference。
3. 新增测试：delegate 第一次抛 `new A2AError(...)`，第二次返回 `Task`，断言第二次能执行且 delegate 调用两次。

### P2: example 脚本即使补齐依赖，也默认不会真正运行 E2E tests

位置：

- `examples/agent-runtime-a2a-llm-e2e/pom.xml:23-30`

```powershell
./mvnw -f examples/agent-runtime-a2a-llm-e2e/pom.xml test
```

```text
Tests are skipped.
BUILD SUCCESS
```

修复建议：

- 如果这个 mixed example 已不再作为默认验证面，README 和脚本名应避免承诺 “runs the E2E suite”。

### P2: 显式开启 example tests 后，`.env.ollama.example` 与硬编码默认值测试冲突

```powershell
  Where-Object { $_ -match '^\s*[^#].*=' } |
  ForEach-Object {
    $k,$v = $_ -split '=',2

./mvnw -f examples/agent-runtime-a2a-llm-e2e/pom.xml test "-DskipTests=false"
```

结果：29 tests，2 failures。

- `SampleLlmGatewayFlipTest.defaultPathKeepsDirectUpstreamAndGatewayOff`
- `SampleLlmGatewayFlipTest.flipRoutesEveryFrameworkThroughTheGatewayWithTheMintedToken`

失败原因：

- `.env.ollama.example` 把 `SAA_SAMPLE_OPENJIUWEN_API_BASE` 设置为 `http://localhost:11434/v1`。
- `SampleLlmGatewayFlipTest.java:26` 和 `:45` 仍硬断言 `http://localhost:4000/v1`。

1. 对 “默认值测试” 使用 `ApplicationContextRunner.withSystemProperties(...)` 或清空相关 env 的隔离策略，避免被 caller env 污染。
2. 或把断言改为验证配置关系，而非硬编码默认 endpoint：
   - raw path 下 `sample.openjiuwen.api-base == SAA_SAMPLE_OPENJIUWEN_API_BASE or default`
   - gateway path 下 `agent-runtime.llm.gateway.aliases.sample-llm.base-url == raw upstream base`
3. 在脚本 CI 路径中增加一条真实验证命令，确保 `.env.ollama.example` 不只是配置文件存在，而是能驱动 suite 完成。
