# 场景视图增补：agent-service L1 跨模块交互场景

## 背景与动机

原提案 [`2026-05-22-agent-service-l1-expansion-proposal.cn.md`](./2026-05-22-agent-service-l1-expansion-proposal.cn.md) 的场景视图（§2.1~§2.5）从 agent-service 单模块视角描述业务运作场景。然而 architecture codex 已明确定义六大模块的依赖方向与契约边界，模块间的协作行为需要在场景层面显式展开，否则：

- 集成验收时不知道在模块边界上该断言什么；
- 半白盒测试缺少锚点——知道模块关系但不知道哪个场景哪条链路该测什么。

因此补充六个跨模块交互场景，每个场景聚焦一条明确的跨模块链路，标明涉及的模块和架构铁律约束。

---

### 2.6 Client→Bus→Service 请求准入与路由场景 (Ingress Routing)

- **串联模块**：agent-client → agent-bus → agent-service
- **典型链路**：
  1. 外部应用通过客户端构造运行请求（携带租户标识、幂等键、追踪标识），提交运行任务。
  2. 客户端不直连服务端，而是向总线提交请求。总线作为唯一入口，执行准入评估和过载检测——正常时接受路由，过载时拒绝或延迟排队。
  3. 总线将已准入的请求路由至服务端的北向接口。
  4. 服务端校验认证、租户、幂等性后，创建 Run/Task/Session，立即返回受理回执（不等待引擎执行）。
- **架构铁律约束**：
  - **A-02**：客户端绝不直接调用服务端——此场景的核心验证点是 client→bus→service 的路由路径完整性和隔离性。
  - **A-10**：请求必须携带租户标识。
  - **A-11**：请求必须有幂等键和追踪标识。
  - **A-03**：总线不解释 Run 内部语义——总线只做准入、路由和通道隔离，不解析执行信封或状态机逻辑。
- **适用场景**：所有 Run 提交的必经入口，是端到端链路的起点。验证此场景可确保入口隔离和准入机制生效。
- **Wave 可行性**：W1 可测。

### 2.7 Service↔Engine 执行委托与中断回调场景 (Execution Delegation)

- **串联模块**：agent-service ↔ agent-execution-engine
- **典型链路**：
  1. 服务端构造执行请求（含引擎类型、结构化任务定义、租户标识），提交给执行引擎。
  2. 执行引擎根据引擎类型**严格匹配**对应的执行器——已知类型匹配到对应执行器处理；未知引擎类型必须报错，**禁止静默 fallback 到其他执行器**。
  3. 执行器执行计算——不持有业务状态，不直接访问数据库或发起网络调用。
  4. 执行完成时，引擎返回执行结果（含成功/失败状态和状态增量）；需要挂起时返回强类型中断信号（中断原语含中断类型枚举，非自由文本描述）。
  5. 服务端收到结果后，经状态机校验状态转换合法性，更新 Run/Task/Session 状态。
- **架构铁律约束**：
  - **A-04**：服务端是 Run/Task/Session 状态变更唯一权威——引擎返回结果后，状态更新只能由服务端完成。
  - **A-05**：引擎禁止依赖服务端——引擎不引用服务端的实体、仓储等内部实现。
  - **A-06**：引擎类型必须严格匹配，禁止静默 fallback——这是服务端→引擎边界最关键的断言。
- **适用场景**：平台中心模式下的主执行链路。验证此场景可确保执行委托的契约纯洁性和状态权威的不分裂。
- **Wave 可行性**：W1 可测。

### 2.8 Service→Middleware 策略注入场景 (Hook Interception)

- **串联模块**：agent-service → agent-middleware
- **典型链路**：
  1. 服务端在关键执行点（如工具调用前、模型推理后、错误发生时、挂起前、恢复后）触发中间件调度。
  2. 调度上下文携带租户标识、追踪标识、执行阶段和元数据——**不携带服务端内部实体**，保证中间件接口纯净。
  3. 已注册的中间件链按顺序执行，每个中间件返回处理结果：放行继续、拒绝失败、短路返回。
  4. 安全类中间件必须优先于成本和观测类中间件执行。
  5. **中间件不推进 Run 状态机**——状态变更仍由服务端的状态机完成。
- **架构铁律约束**：
  - **A-07**：中间件是被动策略层，不拥有主流程调度权——只能返回处理结果，不能直接推进 Run 状态机。此场景的核心验证点是中间件的被动性。
  - **A-09**：跨模块交互必须通过标准契约——中间件的调度上下文不携带服务端内部实体，保证契约纯洁。
- **适用场景**：PII 脱敏、工具鉴权、成本归因、token 统计、熔断策略等横切治理。验证此场景可确保策略注入不破坏主流程控制权归属。
- **Wave 可行性**：W1 可测。

### 2.9 Service↔Service 多智能体对等通信场景 (A2A P2P Communication)

- **串联模块**：agent-service → agent-bus → agent-service
- **典型链路**：
  1. 智能体 A 在执行过程中需要与智能体 B 协作，向总线查询目标智能体的可达信息。
  2. 总线作为路由中介，告知智能体 A 下一跳应该与哪个智能体服务建立通信连接（拓扑发现与路由解析）。
  3. 智能体 A 获知目标地址后，与智能体 B 建立**点对点直接通信**，实现多智能体之间的 P2P 协作——后续的消息交互在两个服务端之间直接进行，不再经过总线中转。
  4. 协作完成后，智能体 B 将结果回调给智能体 A，A 恢复被挂起的任务继续执行。
  5. 整个过程中，总线负责初始的路由解析（告知下一跳），但不参与后续的数据中继——服务端之间建立通信后，总线退居控制面角色（处理取消、超时、异常等控制信号）。
- **架构铁律约束**：
  - **A-02**：所有客户端到服务端的请求仍必须经总线入口，但服务端之间的 P2P 通信是服务端之间的直接交互。
  - **A-03**：总线不解释 Run 内部语义——总线只做路由解析和拓扑发现，不解析协作请求的业务内容。
  - **A-04**：每个服务端仍然是自身 Run/Task/Session 的状态权威——智能体 A 挂起后恢复的状态转换由 A 自己的状态机完成，B 不能直接写 A 的状态。
- **适用场景**：多智能体协同编排、主智能体拆分子任务分发到专业智能体、跨团队智能体互联。验证此场景可确保 P2P 通信的路由发现正确、通信建立可靠、状态权威不分裂。
- **Wave 可行性**：W2。

### 2.10 Client→Bus→Service 工具与技能注册管理场景 (Tool & Skill Registration)

- **串联模块**：agent-client → agent-bus → agent-service
- **典型链路**：
  1. 客户端通过总线向服务端注册任务执行所需的工具集和技能集——声明当前智能体在执行任务时可以使用哪些工具（如数据库查询、API 调用、代码沙箱）和技能（如特定领域的推理能力、知识检索能力）。
  2. 服务端校验注册请求的合法性和完整性（工具定义格式、技能描述规范、租户权限），持久化注册信息，并关联到对应的智能体实例或任务模板。
  3. 客户端可以后续**动态更新**已注册的工具和技能——新增工具、移除工具、更新工具的参数定义或版本、升级技能描述等。
  4. 当任务执行时，服务端将当前已注册的工具集和技能集注入到执行引擎的上下文中，引擎根据注入的定义决定何时调用哪个工具，但引擎本身不持有工具注册表——工具注册的权威在服务端。
  5. 工具和技能的注册与更新是运行时行为，不影响正在执行中的 Run（已加载的工具集不受后续更新影响，直到下一次任务创建时生效）。
- **架构铁律约束**：
  - **A-02**：客户端的注册请求仍必须经总线入口，与 Run 提交走同一路径。
  - **A-04**：服务端是工具和技能注册信息的权威——引擎只是消费注入的工具定义，不自己维护注册表。
  - **A-10**：注册请求必须携带租户标识，工具和技能的注册范围限定在租户边界内。
  - **A-11**：注册和更新操作必须有幂等语义——重复提交相同定义不应产生重复注册。
- **适用场景**：智能体部署时初始化工具集、运行时动态扩展或缩减能力、A2A 协作中声明自身能力供其他智能体发现。验证此场景可确保工具和技能的管理权归属正确、动态更新机制可靠、租户隔离有效。
- **Wave 可行性**：W1 可测。

### 2.11 Bus 三通道隔离与控制优先场景 (Channel Isolation)

- **串联模块**：agent-bus 内部（影响 client→service 和 service→service 的所有链路）
- **典型链路**：
  1. 总线内部维护三条逻辑隔离的通道：
     - **控制通道**：承载取消、恢复、紧急策略变更等控制信号。不可丢弃、不可延迟。
     - **数据通道**：承载 Run 提交、结果回调、轨迹上报等数据载荷。可背压、可延迟。
     - **节拍通道**：承载心跳、指标上报等周期性信号。旧节拍可丢弃。
  2. 高负载场景下，数据通道拥堵时，控制通道的取消信号仍能及时投递——**控制通道不得被数据通道挤占**。
  3. 节拍通道的旧脉冲可丢弃——当前周期的心跳未处理时，新心跳已到达，旧心跳被覆盖而非排队。
  4. 服务端的状态权威与总线的通道隔离协同：总线投递取消控制信号到服务端 → 服务端校验当前 Run 是否非终态 → 合法则转换到取消状态；非法（已终态）则返回错误。
- **架构铁律约束**：
  - **A-12**：控制信号优先于数据载荷——此场景的核心验证点是高负载下取消信号不被数据通道阻塞。
  - **A-03**：总线不拥有业务状态机——总线只投递取消信号，状态校验和状态变更由服务端完成。
- **适用场景**：高并发生产环境、紧急取消、策略热变更。验证此场景可确保总线的通道隔离不成为控制信号的瓶颈。
- **Wave 可行性**：W2。
