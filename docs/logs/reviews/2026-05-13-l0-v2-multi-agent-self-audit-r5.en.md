### Challenge 2 — `agent_runtime_kernel` row in architecture-status.yaml is `shipped: true` with vacuous evidence
- **Severity if true:** P0
- **Defect category:** false-premise (evidence laundering through prose)
- **The claim:** v2 line 14: `The spring-ai-ascend W0 runtime kernel is architecturally ready for L0 release.` v2 lines 56–66 enumerate Run/DFA/Orchestration SPI/AppPostureGate as W0 shipped. v2 line 208 promises `All shipped: true capability rows in docs/governance/architecture-status.yaml have: resolvable implementation: paths (Gate Rule 7), resolvable tests: entries (Gate Rule 19) …`.
- **Strongest counter:** The `agent_runtime_kernel` row at YAML lines 45–55 has `shipped: true`, but the structured fields are vacuous: `implementation:` lists only `OssApiProbe.java` and `tests:` lists only `OssApiProbeTest.java`. The actual three tests in `OssApiProbeTest` are (i) `classIsLoadable` — does nothing but `Class.forName`; (ii) `probeReturnsNonNullString` — asserts `.startsWith("agent-runtime W0-U2 probe:")`; (iii) `temporalGetVersionShapeReturnsMinusOne` — asserts a sentinel of an unreachable invocation. Zero of these tests exercise `Run`, `RunStateMachine`, `SyncOrchestrator`, `IterativeAgentLoopExecutor`, `InMemoryRunRegistry`, `InMemoryCheckpointer`, or `AppPostureGate` — the actual claimed shipped surface. The kernel claim is smuggled into the **prose** `allowed_claim:` field ("Kernel logic shipped: Run entity, RunStatus DFA, RunStateMachine, SyncOrchestrator …. 101 unit tests GREEN in agent-runtime") which Gate Rules 7/19/24 do **not** parse — those rules check only the structured `implementation:`/`tests:`/`l2_documents:`/`latest_delivery_file:` arrays. So the assertion on v2 line 208 ("resolvable tests: entries (Gate Rule 19)") is mechanically true, but mechanically true on a trivial probe shell while the substantive kernel claim travels untethered through `allowed_claim:`. This is the GATE-SCOPE-GAP shape **inside the very ledger that Rule 24 was supposed to canonise** — Rule 24 checks `tests:` field paths exist, not that those tests actually cover the prose claim adjacent to them.
- **Counter to the counter:** The kernel classes do exist on disk (verified: `RunStateMachine.java`, `AppPostureGate.java`, `SyncOrchestrator.java` are present) and other YAML rows DO list tests for them (`run_state_machine_dfa`, `run_state_machine_active`, `runtime_app_posture_gate` rows). So the kernel is in fact tested — just not through the row that says `agent_runtime_kernel.shipped: true`. One could argue the kernel-row structured fields are intentionally minimal and the work is split across sibling rows. But that defense concedes the structural defect: a `shipped: true` row with `implementation: [OssApiProbe.java]` and `tests: [OssApiProbeTest.java]` advertises that those two files **are** the kernel evidence. A reader running Gate Rule 19 would conclude "kernel tested" from `OssApiProbeTest` alone. The mechanism still launders.
- **Verdict on this challenge:** **WEAKENED** (substantive evidence exists elsewhere) but the row's evidence assignment is misleading and constitutes a **NEW 6th-shape defect** I propose: **EVIDENCE-LAUNDERING-VIA-PROSE-FIELD** — a structured ledger row whose enforceable fields are non-representative of the claim its prose field makes.
- **What v2 should say:** Either (a) expand the `agent_runtime_kernel` row's `implementation:` and `tests:` arrays to enumerate the kernel classes and their tests, and reduce `allowed_claim:` to the structured truth; or (b) split `agent_runtime_kernel` into a `agent_runtime_kernel_probe_shell` row (current contents) and a `agent_runtime_kernel_logic` row that lists the real classes. Add a Gate Rule 30 that compares prose claim tokens in `allowed_claim:` against the structured `tests:` field — if the prose names a class that no listed test class exercises, fail.

### Challenge 3 — "No regression vector" in Migration section
- **Severity if true:** P1
- **Defect category:** unverified-strong-claim
- **The claim:** v2 line 284: `No regression vector. Every v1 capability that was shipped is still shipped; every v1 deferral is still deferred (now with stronger ADR backing).`
- **Strongest counter:** Rule 26's pattern catalog targets release notes; the test asset Rule 26 was built around was **v1**. The pointer in `release_note_shipped_surface_truth_gate.implementation` (and Rule 26's self-tests) now refers to v2's text. v1's archived path still contains the original prose that the regex catalog was tuned against (e.g. specific overclaim patterns about `RunLifecycle`, `RunContext.posture()`, `ApiCompatibilityTest`-as-OpenAPI-snapshot, `AppPostureGate`-under-HTTP-Edge). If v2's prose happens not to trip those specific patterns (because v2 was authored carefully), Rule 26 will PASS even though its discriminating power has not been demonstrated on the new active target. The regression vector is: **the gate that was forged against v1 may no longer be exercised by the active document v2 represents.** Rule 26 may now be a rule that always passes for non-v1 inputs in this repository — a tautology dressed as a gate. The self-tests on synthetic strings still pass, but the real-world prose surface that the rule polices is now silent.
- **Counter to the counter:** The self-tests use synthetic positive/negative payloads, not the v1 file itself, so they remain meaningful regardless of which file is active. And v2 is a genuinely different document, not a regression in v1's claims. Granted. But "no regression vector" is still too strong: the *coverage demonstration* (a real document that would have failed Rule 26 had Rule 26 not existed) has moved from active surface to archive. If a future v3 introduces a new variant of the same overclaim pattern, Rule 26 catches it because the self-tests are synthetic. So this is closer to a *coverage-evidence drift* than a regression. Still, the absolute "no regression vector" overclaims; a softer phrasing is warranted.
- **Verdict on this challenge:** **WEAKENED**.
- **What v2 should say:** Replace `No regression vector.` with `No known capability regression: every v1 capability that was shipped is still shipped; every v1 deferral is still deferred. Rule 26's discriminating self-tests are synthetic, so coverage of the overclaim catalog does not depend on v1 remaining the active release note.`

### Challenge 4 — "13 review cycles" framing implies completeness
- **Severity if true:** P2
- **Defect category:** scope-rhetoric
- **The claim:** v2 line 7: `Review cycles: 13 passes`. Lines 15–16: each cycle (a) categorises defects, (b) does systematic self-audits, (c) lands gate-enforced prevention.
- **Strongest counter (treadmill argument):** Gate-rule count history is 14→18→23→25→27→29 across the documented cycles. Every cycle since the 7th has produced ≥2 new gate rules. There is no diminishing-returns evidence: cycle 12 produced ADR-0048; cycle 13 produced four more ADRs (0049–0052) and two more gate rules. Stochastically, a 14th cycle would find at minimum one new defect class — there is no observed asymptote. Declaring "L0 architecturally ready" at cycle 13 is a stopping decision driven by calendar (one calendar day), not by a measured convergence criterion.
- **Counter to the counter (genuine convergence argument):** The cycles HAVE produced a stable canonical lens — the 5-shape defect model. Cycles 8–13 mostly add new instances within the **same** shapes (REF-DRIFT, HISTORY-PARADOX, PERIPHERAL-DRIFT, GATE-PROMISE-GAP, GATE-SCOPE-GAP); they do not invent new shapes. That IS asymptotic behaviour at the meta-level. The kernel surface (Run/DFA/SPI/AppPostureGate) has not changed in the last 6 cycles — only the ledger and prose surfaces have. So the kernel itself is converged; what continues to drift is the *description* of the kernel, which is exactly what one would expect at L0.
- **Verdict on this challenge:** **SURVIVED** (with reservation). The framing is honest in shape (asymptote at meta-level), but the absence of an *explicit stopping criterion* is a gap. v2 should publish a falsifiable definition of "L0 ready" — e.g. "a cycle produces zero new gate rules AND zero new ADRs AND zero new constraint rows" — and note that cycle 13 did not meet it.
- **What v2 should say:** Add a line under Executive Summary: `Stopping criterion: L0 will be declared frozen when one consecutive review cycle produces zero new gate rules, zero new ADRs, and zero new §4 constraint rows. Cycle 13 did not meet this criterion (added Gate Rules 28–29, ADRs 0048–0052, §4 #46–#50). v2 is therefore released as a checkpoint, not a freeze.`

### Challenge 5 — Freeze-marker semantics are one-way
- **Severity if true:** P1
- **Defect category:** drift-vector
- **The claim:** v2 implicitly relies on Rule 28's exemption for `Historical artifact frozen at SHA` to keep v1 from triggering baseline-truth failures (verified: v1 contains the marker; v2 does not).
- **Strongest counter:** Rule 28 only checks the **archived** path. There is no invariant preventing v2 itself from being edited 4+ times today. The v1 lesson — "frozen" marker did not prevent further edits — is unaddressed for v2. The user's task description explicitly asks: "what stops the same thing from happening to v2?" Answer from the code: nothing. Rule 28 only validates that baseline counts in active release notes match the canonical YAML; it doesn't track edit-frequency or require an edit-lock declaration. So v2 has the *same drift vector* v1 had, minus the contradicted-self problem (because v2 does not currently claim to be frozen).
- **Counter to the counter:** Rule 28 actually MAKES v2 self-correcting in a way v1's freeze marker did not: if v2 is edited and its baseline counts are touched without updating the canonical YAML, Rule 28 fails the gate. So v2 is governed by **content invariant** (baseline must match YAML), not by **edit-count invariant** (freeze marker forbade edits). The content-invariant approach is stronger because it admits legitimate edits while catching the drift the freeze marker was supposed to prevent. v1's defect was using a freeze marker (an edit-count invariant) but then editing — v2 doesn't make that claim, so the failure mode doesn't recur.
- **Verdict on this challenge:** **SURVIVED** (with proposed strengthening).
- **What v2 should say:** Add a short Invariants section near the top: `This release note is governed by Gate Rule 28 (release_note_baseline_truth): any edit to the baseline table requires a corresponding edit to docs/governance/architecture-status.yaml or the gate blocks the commit. There is no freeze marker; the document is intentionally amendable as long as the baseline-truth invariant holds. To declare a true freeze, add Historical artifact frozen at SHA <sha> AND archive the file under docs/archive/.` Propose a Gate Rule 30 (`active_release_note_uniqueness`): there must be exactly ONE file under `docs/releases/` whose first heading begins with `# spring-ai-ascend L<digit> Architecture Release` — preventing v3 from co-existing with v2 without an explicit supersession step.

### Challenge 6 — "C/S protocol … all named at L0 contract level" is honest architectural completion
- **Severity if true:** P2
- **Defect category:** scope-rhetoric
- **The claim:** v2 lines 88–121 enumerate C/S Dynamic Hydration, Workflow Intermediary, Memory Ownership Boundary, Skill Topology Scheduler with detailed type names (`TaskCursor`, `HydrationRequest`, `SyncStateResponse`, `BackpressureSignal`, `PermissionEnvelope`, etc.). v2 line 22 calls them "named at L0 contract level". v2 line 95 admits "Java types and wire bindings deferred to W2+".
- **Strongest counter:** Naming type names in an ADR is not the same as freezing a contract. A contract has a signature (method + args + return type), a wire format, and conformance tests. None of the C/S, Workflow Intermediary, Memory Ownership, or Skill Topology types exist as Java code in `src/main/`. There are no `.java` files for `TaskCursor`, `HydrationRequest`, `SyncStateResponse`, `WorkflowIntermediary`, `DelegationGrant`, `BidRequest`, `BidResponse`. So "named at L0 contract level" reduces to "named in a markdown ADR document". That is documentation, not contract.
- **Counter to the counter:** A contract level OF NAMING is a real deliverable distinct from implementation: it commits the project to a specific decomposition. Once `TaskCursor` is named with that meaning, a future implementation cannot rename it without ADR amendment. Naming-as-commitment is what ADRs are for, and v2 is honest about the deferral in the Known Limitations section (line 265: "C/S protocol types … all design-only … Java types and wire formats deferred to W2+"). The architectural completion claim is therefore qualified, not absolute.
- **Verdict on this challenge:** **SURVIVED** with risk. The framing is defensible because v2 discloses the deferral in Known Limitations. But the Executive Summary line 22 should match the Known Limitations line 265 honesty level.
- **What v2 should say:** Replace `named at L0 contract level` with `named at L0 design-contract level (ADR text; Java types and wire formats deferred per Known Limitations)` on every occurrence in the Executive Summary and the four section headers under Architectural Commitments. Avoid the reader having to reach line 265 to discover that "contract" means "ADR prose".

### Challenge 7 — Weakest claim broken: "byte-level parity" between PowerShell and bash gate scripts
- **Severity if true:** P2
- **Defect category:** unverified-strong-claim
- **The claim:** v2 line 74: `Architecture-sync gate | 29 active rules on PowerShell + bash with byte-level parity`.
- **Strongest counter:** "Byte-level parity" between PowerShell and bash scripts is provably false — the two languages have different syntax. The two scripts cannot be byte-identical and both run. What is presumably meant is *behavioural parity* (same rules, same pass/fail decisions on same inputs). The phrase as written is a false claim that no reader checks because they parse it charitably. Adversarially, any literal interpretation of "byte-level parity" fails inspection: `.ps1` files use `$variable` and `if ($x)`; `.sh` files use `$variable` and `if [[ $x ]]`. The two files cannot be the same bytes.
- **Counter to the counter:** "Byte-level parity" is an idiom in some testing communities meaning "byte-level identical outputs on byte-level identical inputs". But that is jargon, and v2 is a release note for external readers.
- **Verdict on this challenge:** **BROKEN** (literal reading).
- **What v2 should say:** Replace `byte-level parity` with `decision parity (same rule set, same pass/fail verdict on same inputs; cross-checked by self-tests under both interpreters)`.
