## 1. 项目定位 (Project Positioning)
`spring-ai-ascend` 是一个深度适配华为鲲鹏与昇腾算力体系、背靠 Java 与 Spring 框架标准、且开放兼容各类主流开源智能体框架的企业级开源智能体 (Agent) 开发与运行时工具底座。它为企业开发者提供一套高可靠的基础设施，使其能够快速构建和部署支持长周期运行与群智协同的数字员工应用；作为一个开源的使能工具项目，它专注提供框架与运行时支撑。

## 2. 设计理念 (Design Philosophy)
*   **业务-系统界面分离设计 (Business-System Interface Separation)**：
    系统与业务的集成边界被严格划定在“任务层”。业务系统无需关心智能体底层的思考轨迹与工具调用细节，只需关注中间的阶段性成果；智能体也无需全盘掌握连续的业务长流程或触碰核心业务隐私，仅需聚焦当前被分配的具体任务要求。这种双向的界面分离，实现了企业平台架构与具体业务的彻底解耦，避免了平台陷入中心化瓶颈或陷入无底洞的定制化开发。
*   **多轨智能体总线设计 (Multi-Track Agent Bus Design)**：
    平台底层采用智能体控制流、数据流和节律流（心跳/时间轴）多轨物理分离的总线架构。这一设计从根本上避免了在多智能体密集协作，或是用户输入复杂图文、视频等多模态大体积数据时可能造成的业务阻塞和拥堵，为企业级智能体应用的大规模、高并发落地奠定了坚实的基础能力。
*   **依赖中介的智能体中断互访设计 (Intermediary-Dependent Interrupt & Interaction)**：
    平台允许智能体在轨迹执行乃至更细的上下文粒度上，随时接受来自其他智能体或真实用户的中断请求。通过这种中介调度机制，系统能够支撑智能体在运行时的实时方向调整、指令中止、乃至安全状态回滚，精准满足企业在复杂真实场景下的核心风控与干预诉求。
*   **面向异构智能体框架的开放兼容 (Open Compatibility for Heterogeneous Frameworks)**：
    平台架构不追求封闭的单极生态，而是允许动态装载异构的智能体框架运行时。支持在“刚性”（如严谨的 SOP 流程图）与“柔性”（如大模型自由探索流）智能体执行模式之间进行无缝切换和嵌套调用。这一理念不仅保护了企业现有的存量智能体资产，更赋予了系统面向未来开放诉求的灵活适配与可组合能力。
*   **面向开发者的全生命周期支撑 (Developer-Centric Full Lifecycle Support)**：
    作为一款工具底座，平台将开发者的体验置于核心位置。在提供基础运行时的同时，平台内置或无缝对接丰富的开发调试工具（用于编排与纠错）、运维可观测工具（用于监控底座与算力健康度）以及运营可视化工具（用于分析智能体效能与轨迹），全方位降低企业开发者的使用门槛与排障成本。

## 4. L0 级逻辑视图 (L0 Logical View)
系统逻辑被严格解耦为六大核心模块，清晰划定了业务应用、执行环境与平台底座的职责边界：

1.  **智能体客户端 (Agent Client)**：面向客户业务开发者，可集成至业务应用 IT 系统或企业员工的 PC/移动端应用。负责智能体的静态业务配置（可操作工具、技能等）；动态的任务要求下发、完成校验与进度管理；维护业务相关的知识体系与语义约束；以及业务应用端的环境观测与操控。
2.  **智能体运行时 (Agent Runtime)**：面向客户业务开发者，建议直接部署使用，同时支持二次开发。负责响应式的 APP 服务化封装；支持 A2A 的服务暴露，以及作为客户端调用其他智能体服务；全生命周期管理智能体会话与任务；向下集成中间件；兼容适配异构框架开发的存量智能体。
3.  **智能体执行引擎 (Agent Execution Engine)**：面向客户业务开发者。支持刚性约束的 Workflow（支持底码编排开发）与动态的 AgentLoop（支持配置化与钩子挂接开发）双模态执行。建议采用开源智能体框架（如 `openJiuwen`）或任意主流开源框架进行开发与对接。
4.  **智能体演进层 (Agent Evolution Layer)**：面向客户业务开发者。通过构建数据飞轮收集完备的运营数据，提供双轨制的智能体自学习与持续迭代机制。
5.  **智能体总线 (Agent Bus)**：面向客户平台开发者，建议直接部署使用，支持二次开发。负责客户端与服务端的访问路由、服务端之间的 A2A 跨节点调用、以及承载智能体自触发的心跳节律。
6.  **智能体中间件 (Agent Middleware)**：面向客户平台开发者，建议直接部署使用，支持二次开发。提供标准化的公共底层服务，包括智能体记忆系统、智能体技能中心、以及安全隔离的智能体沙箱服务等。

## 5. L0 级开发视图 (L0 Development View)
本系统的代码工程结构与防腐边界遵循严密的模块划分与仓储矩阵规范，以确保企业级底座的高内聚与低耦合：

### 5.1 仓储矩阵与技术栈规划
为保持开源社区的活力并兼顾异构生态的优势，核心逻辑模块建议以独立仓的形式托管于开源社区（如 `openJiuwen`）：
*   **智能体核心执行引擎 (agent-core-java)**：系统的纯净发动机（Pure Java 构建）。负责 DFA 状态机与图执行等底层调度。严禁包含任何 Spring 框架层面的强绑定注解，仅对外暴露核心 SPI。
*   **智能体运行时 (agent-runtime-java)**：系统的北向服务大门。采用 Spring 技术栈构建，负责将 Core 层的纯 Java 逻辑封装为企业级的 REST/gRPC 服务，处理租户拦截与安全控制。
*   **智能体客户端 SDK (agent-client-sdk-java)**：提供面向业务开发的无侵入集成工具包，支持零外部依赖（Zero-Dependency）引入各类业务 IT 系统。
*   **智能体总线 (agent-bus-java)**：承载通信路由与心跳节律的独立枢纽。支持多级实现（从开发态的内存级总线到生产态对接 Kafka/Redpanda 等分布式组件）。
*   **智能体演进层 (agent-evolution-python)**：专属于算法与自学习的**异构生态仓**。利用 Python 在机器学习、强化学习与微调（Fine-tuning）领域的生态垄断优势，承接重度计算、轨迹分析与自动提示词优化。物理上天然实现离线分析与在线业务的隔离。
*   **智能体中间件模块**：暂不构建独立仓，通过在 Runtime 层引入成熟的开源组件（如 `Spring AI` 对接模型网关，`PostgreSQL` + `pgvector` 作为持久化运行库与记忆底座）实现。

### 5.2 开发态防腐红线与依赖原则
1.  **单向引力与内核纯洁性原则**：`Agent Client` 和 `Agent Runtime` 可以单向依赖 `Agent Core`，但 `Agent Core` 绝对处于依赖树的最底层，严禁反向引入任何上层（如接入层或具体业务场景）的代码。
2.  **中间件 SPI 倒置原则**：核心逻辑不可直接耦合具体的存储或中间件实现。`agent-core-java` 只负责定义存储契约（如 `Checkpointer SPI`），具体的持久化适配（如 Postgres 实现）必须在运行时（Runtime）注入。
3.  **多语言协同红线 (Polyglot Collaboration Boundary)**：执行引擎（Java）与演进层（Python）之间绝对禁止内存级共享或私有库依赖。两者必须且只能通过智能体总线（Agent Bus）以及语言无关的数据序列化契约（如 Protobuf 或标准 JSON Schema）进行状态同步与逆向采样触发。

## 6. L0 级进程视图 (L0 Process View)
进程视图聚焦于系统在运行时的并发模型、跨模块通信范式以及阻塞控制策略，旨在确保系统在面对多智能体海量协同与长周期任务时的绝对健壮性与高吞吐：

### 6.1 C/S 异步交互与游标流转范式
*   **绝对异步的客户端通信**：`Agent Client` 与 `Agent Runtime` 的交互禁止采用传统的同步阻塞模式。客户端提交长周期任务后，系统必须立即返回带有生命周期的任务游标（Task Cursor）。
*   **多态反馈通道**：针对过程状态，客户端可通过 SSE (Server-Sent Events) 建立轻量级子流获取实时推演过程；针对阶段性结果或需人类介入（Human-in-the-loop）的阻塞节点，系统必须通过 Webhook 异步回调唤醒客户端，彻底摒弃客户端的长连接死等。

### 6.2 跨语言生态的影子流量消费模型
*   **主链路与演进层隔离**：Java 执行引擎产生的思维链轨迹、工具调用耗时与中间态结果，必须以“事件溯源（Event Sourcing）”的形式向总线的数据轨异步抛出。
*   **旁路消费闭环**：Python 构建的 `Agent Evolution Layer` 作为总线上的独立旁路消费者，以接收“影子流量（Shadow Traffic）”的方式进行离线清洗、强化学习打分与知识图谱构建。严禁在主执行链路上同步等待算法层的计算反馈，确保核心引擎处理延迟不受大模型微调等重负载算法拖累。

### 6.3 外部 I/O 调用的绝对非阻塞铁律
*   **响应式与虚拟线程底座**：针对 `Agent Middleware` 层中极易产生长延迟的外部调用（如大模型推理生成、大型向量库检索、沙箱脚本执行），运行时必须采用响应式编程模型（Reactive）或轻量级协程/虚拟线程（Virtual Threads）。
*   **禁止 I/O 独占**：在等待外部 API 返回的数十秒真空期内，底层的 OS 级计算线程必须被强制释放并让渡给其他 Agent 实例使用，从物理层面杜绝因外部网络 I/O 挂起导致的系统级并发瘫痪。

### 6.4 引擎内部调度与防护基线
*   **工作流中介的 Pull 模式背压**：摒弃总线的暴力 Push 派发。总线仅将执行意图投递至计算节点前置的中介信箱；Agent 计算引擎根据自身当前的内存水位与并发配额主动 Pull 获取任务，形成天然的保护性背压缓冲。
*   **节拍接管与进程自毁**：严禁在业务代码中使用物理线程休眠（如 `sleep()`）。所有长周期休眠必须转换为声明式挂起，交由总线层面的节拍引擎（Tick Engine）接管。休眠期间，执行该 Agent 的计算进程必须立即释放或自毁（Chronos Hydration），直到总线唤醒脉冲到达方可重新水合拉起。
*   **三轨物理信道隔离**：跨服务内部通信强制切分为三条独立物理信道。强控制流（如 PAUSE, KILL 指令）拥有最高优先级的独立带外通道；数据计算流（如文生视频文件流转）走带内重载通道；心跳/节拍流负责维持生存状态脉冲。三轨绝对隔离，防止任何单一类型的网络拥堵瘫痪全局。

## 7. L0 级物理视图 (L0 Physical View)
物理视图关注系统组件在生产环境下的物理拓扑、隔离级别与资源分配边界，是保障系统高可用、高安全与防雪崩的最后一道防线：

### 7.1 分布式五平面部署拓扑 (Five-Plane Deployment Topology)
物理部署被严格划分为相互隔离的五个平面，确保不同特性的负载互不干扰：
*   **边缘接入平面 (Edge Access Plane)**：`Agent Client` SDK 所在的物理位置。它可以嵌入在企业内网的业务服务器中，或直接运行在终端设备上。作为整个底座的物理入口，其部署特征是极度轻量、零状态、并直接面对网络边缘的安全防护。
*   **计算与控制平面 (Compute & Control Plane)**：`Agent Runtime` 与 `Agent Execution Engine` 所在的区域。此平面的计算节点必须是无状态的（Stateless）容器化集群（如 K8s Pods），负责处理高并发的指令解析与状态机流转，支持秒级的水平弹性扩缩容。
*   **总线与状态枢纽平面 (Bus & State Hub Plane)**：`Agent Bus`（作为集群独立的物理 Broker 集群，如 Kafka/Redpanda）与 `Agent Middleware`（如 PostgreSQL, 向量数据库）构成的重数据平面。此平面是系统状态唯一的物理锚点，要求部署在具备极高 I/O 吞吐与持久化保障的独立存储计算节点上。
*   **沙箱执行平面 (Sandbox Execution Plane)**：系统底线的物理安全隔离区。所有由大模型动态生成的代码（如代码解释器输出）或未验证的第三方工具，**绝对禁止**在计算控制平面执行，必须被调度至独立的 Serverless 容器组或 Firecracker 微虚拟机等一次性销毁的物理沙箱中运行。
*   **演进与分析平面 (Evolution Plane)**：`Agent Evolution Layer`（Python 生态）所在的独立离线或准实时计算集群。为了防止模型微调或海量数据清洗拖垮在线交易的 CPU 与内存，此平面必须部署在独立的物理节点组（如专享 GPU Node Pool）中，通过跨平面订阅总线的旁路流量进行工作。

### 7.2 多租户与数据资产的物理存储红线
作为企业级底座，应用代码层的隔离逻辑被视为“不安全的”。多租户的数据隔离必须下沉至存储引擎的物理或机制底座。例如，所有持久化至数据库的 Agent 记忆与轨迹，强制启用底层引擎的行级安全策略（Row-Level Security, RLS）或采用独立的 Schema/Database 隔离，确保即使运行时代码出现越权漏洞，也无法在物理存储引擎层窃取跨租户资产。

### 7.3 基于技能拓扑的物理资源仲裁
为防止单一高频技能（如某个慢速的外部 API）耗尽集群整体的连接池与 CPU 资源，系统在物理层面建立全局技能拓扑调度器。当技能并发池满时，调度器会精准挂起（Suspend）且仅排队依赖该技能的特定 Agent 计算进程，将底层的 OS 级线程资源让渡给执行其他轻量推理的 Agent 实例，形成二维度（租户配额 × 全局技能容量）的物理防御网。

### 7.4 代理竞标与物理沙箱权限对齐
跨物理节点的协同必须基于“零信任”假设。
*   **预授权接入**：跨网络发起能力注册的节点必须绑定平台签发的领域权限凭证。
*   **沙箱权限包容原则 (Skill Subsumption)**：总线意图竞标成功后，平台向下发执行权限时，必须将逻辑授权范围 1:1 映射至该节点所处物理沙箱的系统级限制（如网络出站 IP 白名单、CPU 使用上限），确保执行权的逻辑下发不会突破物理沙箱的隔离上限。

## 8. 分层架构与代码贡献守则与流程

### 8.1 4+1 视图共识定义
为确保各架构师与开发团队的设计语言对齐，所有架构文档必须基于“4+1 视图”模型展开：
*   **逻辑视图 (Logical View)**：关注领域划分、实体模型、模块接口与系统职责边界。
*   **开发视图 (Development View)**：关注代码组织、包结构、模块依赖关系（禁止反向依赖）与防腐层。
*   **进程视图 (Process View)**：关注并发模型、同步/异步通信范式、数据流/控制流/节律流的时序与阻塞控制。
*   **物理视图 (Deployment View)**：关注部署拓扑、计算节点分配、存储物理隔离级别与运行环境。
*   **场景视图 (Scenarios View / +1)**：使用核心业务场景或流转链路将上述 4 个视图串联，验证架构设计的可行性与自洽性。

### 8.2 分层架构规范
系统架构严格分为三个递进层级，**每个层级都应当按照 4+1 视图进行展开**：
*   **L0 级 (顶层设计)**：描述系统的全局边界、核心理念与原则。L0 级的 4+1 视图仅定共识与宏观框架。
*   **L1 级 (领域设计)**：在 L0 框架下，对特定核心能力块展开子系统级的 4+1 视图设计。
*   **L2 级 (技术详设)**：面向具体特性和需求用例展开的详细物理设计与执行设计。L2 级 4+1 视图可根据具体特性做“不相关视图的缺省（Omit）”，聚焦核心实现细节。

### 8.3 架构文档流转与评审机制
*   **按阶段发布与冻结**：架构文档应当按阶段（Phase）Release 发布，一旦发布结版，绝对不允许直接修改。
*   **超前设计归档隔离**：任何超出当前阶段的架构设计（无论属于 L0/L1/L2 哪一层级），都应当作归档（Archive）处理，避免干扰当前阶段的工程实现。
*   **架构变更评审流**：所有的架构修改意见必须首先进入 `docs/reviews/` 目录形成提案，且提案**必须明确指出涉及的架构层级与受影响的视图范围**。
*   **修改核准**：变更提案必须经过总架构师（或其授权的 AI 架构助手）的严格评审。评审通过后，方可由授权人更新当前阶段的正式架构文档。

### 8.4 代码贡献守则 (Code Contribution Guidelines)
所有合入主干的 PR 必须满足以下刚性约束：
1.  **代码常规检查**：必须通过所有单元测试、代码规范（Linting）与基础静态检查。
2.  **L0 原则冲突校验**：代码逻辑不得违背 L0 级架构宣告的核心原则与设计理念。
3.  **L1 边界范围校验**：包引用、依赖关系与模块通信必须严格遵守 L1 领域视图划分的防腐边界与可见性控制。
4.  **L2 详设一致性校验**：代码级类结构、接口定义与时序必须与 L2 级技术详设保持高度一致。
5.  **契约即代码 (Code-as-Contract)**：相关的架构级约束必须在 `docs/governance/enforcers.yaml` 中绑定自动化守护测试（ArchUnit 或验证脚本）。无自动化约束保障的代码严禁合入。
