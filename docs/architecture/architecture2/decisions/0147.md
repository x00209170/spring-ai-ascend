# 0147. Structurizr workspace closure as the architecture authoring root, with programmatic mounting

Date: 2026-05-27

## Status

accepted

## Context

The current architecture-authority chain (per ADR-0068 + Rule G-1.b)
derives `docs/governance/architecture-graph.yaml` from four
authoritative YAML inputs (`enforcers.yaml`, `principle-coverage.yaml`,
`architecture-status.yaml`, `*/module-metadata.yaml`) plus the
76-yaml ADR corpus and L1 architecture-doc front-matter. The graph
is 620 nodes / 1172 edges spanning 13 node types and 16 edge types.

Two structural problems remain:

1. The L1 feature inventory and function-point semantics are split
   across L1 prose, `architecture-status.yaml#capabilities`,
   `architecture-graph.yaml` capability nodes, and L2 docs. AI
   sessions must load 5+ files before reasoning about a single
   feature's owners, tests, and decision history.

2. The cross-authority parity surface (Rule G-8, Rule G-13)
   remains defensive: every authority surface is internally
   well-formed but the gate must re-prove they agree with each
   other on every change.

An expert proposal at `docs/plans/2026-05-27-structurizr-workspace-authority-remediation-plan.md`
recommends adopting Structurizr DSL as a single authoring root,
using `architecture/workspace.dsl` plus !include / !docs / !adrs
for closure. Independent analysis found three fatal defects in
that proposal (wrong DSL keyword `customElement`, validator
walking the wrong collection, stale upstream repo pointer) and
seven structural gaps including a 612-node hand-authoring burden
that would double drift surface.

Wave 0 spike (this ADR's companion ADR-0148) measured the
technical assumptions empirically against the agent-bus slice.

## Decision

1. Adopt `architecture/workspace.dsl` and its workspace closure
   as the single architecture authoring root, AFTER all 7 waves
   in the replacement plan land. The closure is:

   - architecture/workspace.dsl              (root + views)
   - architecture/profile/                   (Spring AI Ascend Architecture Profile)
   - architecture/features/                  (capabilities, function points, verification — AUTHORED)
   - architecture/docs/L1/                   (L1 narrative descriptions — AUTHORED)
   - architecture/decisions/                 (ADR markdown imports — AUTHORED)
   - architecture/generated/                 (modules, SPI catalog, enforcers, principles,
                                             rules, ADR-graph, surface-classification — GENERATED, never hand-edited)
   - architecture/views/                     (view declarations)

2. The closure is partially generated. The "generated zone"
   (`architecture/generated/*.dsl`) is emitted by Java fragment
   emitters in `tools/architecture-workspace/` from the existing
   authoritative inputs (`*/module-metadata.yaml`,
   `docs/governance/enforcers.yaml`, etc.). Engineers NEVER
   hand-edit the generated zone. The gate enforces byte-identical
   re-emission. This prevents the dual-authoring drift the
   original proposal would have introduced.

3. The "authored zone" (workspace.dsl root, features/, docs/L1/,
   decisions/) is where engineers add new L1 capabilities,
   function points, ADR descriptions, and L1 narrative. This is
   the AI-readable L1 feature/function-point registry the
   migration delivers.

4. The Spring AI Ascend Architecture Profile is normative. It is
   defined in `architecture/profile/profile.yaml` and enforced
   by `tools/architecture-workspace/src/main/java/.../ProfileValidator.java`.
   The validator walks `Model.getElements()` exactly once
   (CustomElement extends Element in structurizr-dsl 6.2.1, so
   walking both Model.getElements() AND Model.getCustomElements()
   double-counts — empirically verified per ADR-0148
   §spike-evidence empirical-finding-1).

5. `docs/governance/architecture-graph.yaml` is preserved during
   the migration as a generated compatibility projection (Wave 4
   deliverable). It is NEVER hand-edited. Authority status moves
   from "single machine-readable index" (Rule G-1.b kernel text
   today) to "generated compatibility projection" (Wave 5
   amendment).

6. The legacy YAMLs (`enforcers.yaml`, `principle-coverage.yaml`,
   `architecture-status.yaml#capabilities`) are sunset gradually
   in Wave 6. Their authority direction flips: they become
   generated projections OF the workspace, not source-of-truth.
   `module-metadata.yaml` and ADR YAMLs are retained as
   operational compat inputs (Maven, ArchUnit) but lose
   authority status.

7. Structurizr dependencies are build-time only. The
   `tools/architecture-workspace/` Maven module is NOT a member
   of the production reactor (root `pom.xml#modules` does NOT
   list it). No production module depends on `com.structurizr:*`.
   Structurizr server is explicitly out of scope.

8. Wave decomposition (per the replacement plan —
   historical session plan, local-only):
     - Wave 0: spike + ADR (this ADR + ADR-0148)
     - Wave 1: tooling foundation (advisory gate)
     - Wave 2: authored zone
     - Wave 3: generated zone (7 fragment emitters)
     - Wave 4: compatibility projection + L1 rendering
     - Wave 5: authority transfer (gate flips advisory→blocking; 14d soak)
     - Wave 6: YAML authority sunset (60d soak)
     - Wave 7: final cleanup (rule retirements + closure ADR)

9. Rule G-1.b is amended in Wave 5 (not before). The new kernel
   sentence becomes:

     "The machine-readable architecture authority MUST be rooted
      in architecture/workspace.dsl and its workspace closure
      (architecture/profile/, architecture/features/,
      architecture/docs/, architecture/decisions/,
      architecture/generated/, architecture/views/). Generated
      projections, including docs/governance/architecture-graph.yaml,
      MUST be built from that closure, never hand-edited, and
      byte-identical on regeneration."

## Consequences

Positive:
- L1 feature/function-point inventory becomes a single
  AI-readable registry rooted at `architecture/workspace.dsl`.
  AI sessions can answer "what L1 function points exist, who
  owns them, which ADRs decided them, which tests verify them?"
  without loading other authority files.
- Cross-authority drift surface shrinks: generated-zone fragments
  are byte-identical-re-emit on every change, eliminating the
  Rule G-8 / G-13 defensive comparisons across multiple yaml
  files.
- Structurizr DSL adoption gives the project free Mermaid /
  PlantUML / static-HTML exports via `structurizr-export` (already
  transitive of `structurizr-dsl:6.2.1`).
- The profile validator is a single Java enforcer; new edge
  types or required properties are added via profile YAML +
  validator update, not via gate-script regex expansion.

Negative / residual:
- Migration is 7 waves, ~15-22 weeks of engineering. Wave 5 has
  a 14-day soak gate; Wave 6 has a 60-day soak gate before
  Wave 7. Rollback after Wave 5 is intentionally expensive.
- Engineers must learn Structurizr DSL syntax (small surface)
  plus the Spring AI Ascend Architecture Profile (~12 tags,
  ~15 relationship types, ~13 required-property classes).
- The original expert proposal's defects (wrong DSL keyword,
  wrong element iteration, stale upstream pointer) are
  individually small but collectively would have caused silent
  failure — they are documented here so future migrations don't
  repeat them.
