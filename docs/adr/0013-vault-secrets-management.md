# 0013. UUIDv7 for surrogate IDs, not snowflake / sequence

**Status:** accepted
**Deciders:** architecture
**Date:** 2026-05-10
**Technical story:** Every persistent record needs a surrogate ID that is safe under multi-tenant write load without a central sequence.

## Context

Every persistent record in the system requires a surrogate ID that is globally unique,
safe for multi-tenant concurrent writes, and suitable for chronological B-tree indexing.
The ID generation strategy affects both write performance and cross-region replication
safety.

## Decision Drivers

- No central sequence: safe for cross-replica and cross-region writes.
- Time-ordered: B-tree pages stay packed for chronological queries.
- Globally unique: safe for cross-region replication without coordination.
- App-generated IDs allow ID assignment before the database write, simplifying outbox patterns.

## Considered Options

1. UUIDv7 (time-ordered) -- no central sequence; B-tree friendly.
2. UUID v4 (random) -- no central sequence; B-tree fragmentation under write load.
3. Postgres bigserial -- sequence becomes a hotspot with read replicas; cross-region contention.
4. Twitter-style snowflake -- requires a central time service or node registry.

## Decision Outcome

**Chosen option:** Option 1 (UUIDv7 generated by app via java.util.UUID + helper), because
it combines global uniqueness with time-ordering to maintain B-tree index density and
removes the central-sequence hotspot that makes bigserial problematic under multi-region load.

### Consequences

**Positive:**
- No central sequence; safe for cross-replica and cross-region writes.
- Time-ordered UUIDs keep B-tree pages packed for chronological access patterns.
- App-side generation simplifies outbox and event-sourcing patterns.

**Negative:**
- Application must generate IDs (slight overhead vs DB-generated defaults).
- UUIDv7 helper is code the team owns and must maintain.

### Reversal cost

high (every table's primary key + every Java record DTO)

## Pros and Cons of Options

### Option 1: UUIDv7 (time-ordered)

- Pro: No central sequence; safe for distributed concurrent writes.
- Pro: Time-ordering keeps B-tree index pages packed.
- Pro: App-side generation simplifies outbox and idempotency patterns.
- Con: Slight overhead versus DB-generated IDs.
- Con: UUIDv7 helper is custom glue owned by the team.

### Option 2: UUID v4 (random)

- Pro: No central sequence; simpler to generate.
- Con: Random distribution causes B-tree fragmentation under sustained write load.
- Con: No chronological ordering; harder to debug event sequences.

### Option 3: Postgres bigserial

- Pro: Simple; DB-native; no custom helper.
- Con: Sequence becomes a hotspot on read replicas and under cross-region write contention.
- Con: Requires DB round-trip before ID is known; complicates outbox patterns.

### Option 4: Twitter-style snowflake

- Pro: Time-ordered and compact.
- Con: Requires a central time service or node-ID registry to avoid collisions.
- Con: Adds operational dependency not justified at v1 scale.
